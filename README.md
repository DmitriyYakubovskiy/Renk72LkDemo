# Renk72LkDemo

**Renk72LkDemo** — демонстрационная версия личного кабинета заявителя для подачи и сопровождения заявок на технологическое присоединение.

Репозиторий содержит упрощённую публичную реализацию системы и предназначен для демонстрации архитектуры проекта, основных пользовательских сценариев и применяемых технических решений.

> **Важно:** данный репозиторий не является полной копией промышленной системы.  
> Рабочая версия личного кабинета используется по адресу [lk.renk72.ru](https://lk.renk72.ru/) и развивается отдельно. Часть функций, интеграций, внутренних сервисов и механизмов обработки данных в демонстрационную версию не включена либо представлена в упрощённом виде.

## Назначение проекта

Система предназначена для автоматизации взаимодействия заявителя с сетевой организацией при подаче заявки на технологическое присоединение.

Через личный кабинет пользователь может:

- зарегистрироваться и войти в систему;
- создать заявку;
- последовательно заполнить разделы заявки;
- указать сведения о заявителе и представителе;
- описать объект технологического присоединения;
- заполнить технические характеристики;
- прикрепить необходимые документы;
- сформировать заявление;
- отслеживать состояние обращения;
- обмениваться сообщениями в рамках заявки.

Демонстрационная версия показывает основные принципы построения такого сервиса, но не охватывает весь функционал промышленной системы.

## Сценарий работы пользователя

### 1. Регистрация

Пользователь создаёт учётную запись, указывая необходимые регистрационные данные.

Для управления пользователями применяется ASP.NET Core Identity.

### 2. Создание заявки

После авторизации пользователь начинает создание новой заявки.

Данные заполняются последовательно по этапам:

1. сведения о заявителе;
2. сведения о представителе;
3. сведения об объекте технологического присоединения;
4. технические характеристики;
5. документы и подтверждение заявления.

Набор и порядок этапов в промышленной версии могут отличаться.

### 3. Загрузка документов

Файлы, прикреплённые к заявке, сохраняются через объектное хранилище MinIO.

### 4. Формирование заявления

На основании заполненных данных отдельный сервис формирует PDF-документ для предварительного просмотра.

### 5. Отправка и сопровождение

После заполнения пользователь может сохранить или отправить заявку и далее отслеживать её состояние.

В демонстрационной версии этот процесс может быть сокращён по сравнению с рабочей системой.

## Аутентификация и роли

В проекте используется ASP.NET Core Identity и cookie-аутентификация.

Основные роли:

- `User` — заявитель;
- `Admin` — администратор демонстрационной системы.

Настройки по умолчанию предусматривают:

- обязательный уникальный email;
- пароль длиной не менее восьми символов;
- наличие цифры в пароле;
- временную блокировку учётной записи;
- сессию продолжительностью четыре часа;
- восстановление пароля по токену;
- проверку статуса блокировки пользователя.

## Работа с файлами

Для хранения вложений используется MinIO.

В объектном хранилище могут размещаться:

- документы заявки;
- подтверждающие файлы;
- сформированные документы;
- служебные вложения.

Данные подключения задаются в разделе:

```json
{
  "MinIO": {
    "Host": "minio:9000",
    "AccessKey": "change_this_value",
    "SecretKey": "change_this_value"
  }
}
```

## Запуск без Docker

### 1. Восстановление зависимостей

```bash
dotnet restore Renk72LkDemo.sln
```

### 2. Настройка базы данных

Создайте базу MySQL:

```sql
CREATE DATABASE renk72lk
    CHARACTER SET utf8mb4
    COLLATE utf8mb4_unicode_ci;
```

Создайте отдельного пользователя:

```sql
CREATE USER 'renk72lk_user'@'localhost'
IDENTIFIED BY 'change_this_password';

GRANT ALL PRIVILEGES
ON renk72lk.*
TO 'renk72lk_user'@'localhost';

FLUSH PRIVILEGES;
```

### 3. Настройка строки подключения

```json
{
  "ConnectionStrings": {
    "DbConnection": "server=localhost;port=3306;database=renk72lk;user=renk72lk_user;password=change_this_password"
  }
}
```

### 4. Применение миграций

```bash
dotnet ef database update \
    --project Renk72Lk.DataAccess \
    --startup-project Renk72Lk
```

Когда `dotnet ef` не установлен:

```bash
dotnet tool install --global dotnet-ef
```

### 5. Запуск приложения

```bash
dotnet run --project Renk72Lk
```

При ручном запуске необходимо отдельно настроить и запустить:

- MySQL;
- MinIO;
- RabbitMQ;
- сервис генерации PDF.

## Управление Docker Compose

Остановить сервисы:

```bash
docker compose down
```

Перезапустить:

```bash
docker compose restart
```

Пересобрать:

```bash
docker compose down
docker compose up --build -d
```

Просмотреть запущенные контейнеры:

```bash
docker compose ps
```


## Возможные проблемы

### Приложение не подключается к MySQL

Проверьте:

- состояние контейнера MySQL;
- строку подключения;
- имя сервера `mysql` при запуске через Docker;
- имя сервера `localhost` при запуске без Docker;
- логин и пароль;
- наличие базы данных;
- результаты healthcheck.

```bash
docker compose logs -f mysql
```

### Не формируется PDF

Проверьте сервис:

```bash
docker compose logs -f pdfgenerator
```

Убедитесь, что:

- контейнер запущен;
- выполнены миграции Laravel;
- доступна база MySQL;
- в контейнере существуют необходимые каталоги;
- приложение использует корректный адрес PDF-сервиса.

### Не загружаются документы

Проверьте MinIO:

```bash
docker compose logs -f minio
```

Откройте панель управления:

```text
http://localhost:9001
```

Убедитесь, что параметры `AccessKey`, `SecretKey` и имя хранилища совпадают с настройками приложения.

### Не обрабатываются сообщения RabbitMQ

Проверьте состояние брокера:

```bash
docker compose logs -f rabbitmq
```

Откройте панель:

```text
http://localhost:15672
```

Проверьте имя очереди, логин, пароль и адрес хоста.

### Занят порт

В стандартной конфигурации используются порты:

```text
80
85
3306
5672
8000
8080
9000
9001
15672
```

Измените внешнюю часть сопоставления портов в `docker-compose.yml`, когда один из них уже используется.

## Безопасность

Конфигурация Docker Compose предназначена для локального демонстрационного запуска и не должна использоваться в production без доработки.

Перед публикацией необходимо:

- заменить стандартные пароли;
- вынести секреты в `.env`;
- не использовать пользователя MySQL `root` для приложения;
- закрыть внешние порты MySQL, MinIO и RabbitMQ;
- ограничить доступ к phpMyAdmin;
- настроить HTTPS;
- использовать безопасные cookie;
- отключить Swagger в production;
- настроить резервное копирование;
- добавить централизованный аудит;
- установить ограничения на загружаемые файлы;
- проверять MIME-типы и содержимое вложений.

Пример `.env`:

```env
MYSQL_ROOT_PASSWORD=replace_with_strong_password

MINIO_ROOT_USER=replace_with_minio_user
MINIO_ROOT_PASSWORD=replace_with_strong_password

RABBITMQ_DEFAULT_USER=replace_with_rabbitmq_user
RABBITMQ_DEFAULT_PASS=replace_with_strong_password
```

Файл `.env` необходимо добавить в `.gitignore`.
